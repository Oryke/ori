# ORI Roadmap

ORI is being built in deliberate stages.

Each stage represents a meaningful capability, not a list of development tasks. The technologies we use may change over time, but the direction of the project should remain consistent.

Our goal isn't simply to build another developer tool. Our goal is to make repository knowledge visible, understandable, and reusable for everyone who contributes to open source.

---

# Phase 0 — Foundation

Status: In Progress

Every meaningful project begins by answering "Why?"

Before writing code, ORI establishes the language, principles, and long-term direction of Repository Intelligence.

This phase focuses on:

- defining Repository Intelligence
- documenting ORI's vision
- establishing the project's philosophy
- creating a shared vocabulary
- designing the project's architecture

Deliverables:

- README
- Vision
- Repository Intelligence
- Roadmap
- Initial specifications

Success means people understand the problem ORI exists to solve.

---

# Phase 1 — Repository Understanding

Status: Planned

Before ORI can help maintainers, it must first learn how repositories communicate information.

This phase explores how repository artifacts relate to one another, including documentation, pull requests, issues, commit history, governance, releases, and contributor activity.

The objective is observation rather than evaluation.

ORI should learn to answer questions such as:

- What information exists?
- Where is important context stored?
- What signals are missing?
- Which areas appear difficult to understand?

Success means ORI can build a structured understanding of a repository without making assumptions about its quality.

---

# Phase 2 — Repository Intelligence Engine

Status: Planned

This phase introduces ORI's reasoning layer.

Instead of simply collecting repository data, ORI begins connecting information across different artifacts.

The objective is to transform isolated repository events into meaningful context.

Examples include:

- connecting discussions to code changes
- tracing architectural decisions
- identifying missing context
- highlighting undocumented knowledge
- surfacing historical reasoning

Success means ORI understands relationships rather than isolated files.

---

# Phase 3 — Actionable Insights

Status: Planned

Knowledge only becomes valuable when it can improve decisions.

During this phase ORI begins producing practical recommendations for maintainers and contributors.

Examples include:

- onboarding improvements
- documentation opportunities
- governance observations
- contributor guidance
- repository health recommendations

ORI should explain why each recommendation exists instead of producing unexplained scores.

Success means ORI helps repositories improve themselves.

---

# Phase 4 — Community Intelligence

Status: Planned

Repositories do not exist in isolation.

This phase explores how knowledge can be shared across communities while respecting each project's independence.

Future possibilities include:

- reusable repository patterns
- shared best practices
- contributor learning pathways
- community knowledge preservation
- cross-project insights

Success means repository knowledge becomes easier to transfer without losing local context.

---

# Phase 5 — Open Repository Intelligence

Status: Vision

The long-term ambition for ORI is to help establish Repository Intelligence as an open concept that other projects, organizations, and platforms can adopt.

ORI should not become the only implementation.

Success means:

- Repository Intelligence becomes a recognized discipline.
- Other tools integrate Repository Intelligence concepts.
- Open source projects benefit from stronger institutional memory.
- Maintainers spend less time repeating history.
- Contributors spend more time building and less time searching for context.

If one day maintainers naturally ask, "What does ORI say about this repository?", this project will have achieved something meaningful.

---

# Guiding Principles

Every phase should move ORI closer to these principles.

- Repository knowledge should be discoverable.
- Context should survive maintainer turnover.
- Decisions should remain understandable.
- Documentation should explain reasoning, not only instructions.
- Repository Intelligence should augment maintainers rather than replace them.
- ORI should remain open, transparent, and community-driven.

---

# A Living Roadmap

This roadmap is intentionally iterative.

New ideas will emerge.

Research will reshape assumptions.

The implementation may change.

The mission should not.

ORI is being built for the long term, and this roadmap will evolve alongside the project.
