# Evidence Model

## Trust Through Evidence

Repository Intelligence should never rely on assumptions alone.

Every observation, insight, and recommendation produced by ORI must be supported by evidence that can be traced back to the repository itself.

The Evidence Model defines what ORI considers evidence, how evidence is represented, and how evidence supports Repository Intelligence.

Its purpose is not to prove that ORI is always correct.

Its purpose is to ensure every conclusion is transparent, explainable, and open to verification.

---

# Design Philosophy

Evidence is the foundation of trust.

A maintainer should never have to wonder:

> "Why did ORI say this?"

Every conclusion should answer that question automatically.

Repository Intelligence is strongest when every observation can be understood, inspected, and challenged using repository artifacts.

---

# What Is Evidence?

Within ORI, evidence is defined as:

> Repository information that directly supports, explains, or justifies an observation or insight.

Evidence is not interpretation.

Evidence is the information from which interpretation is made.

---

# Sources of Evidence

ORI recognizes multiple forms of repository evidence.

## Documentation Evidence

Knowledge captured in written documentation.

Examples include:

- README
- Architecture documents
- Contribution guides
- Decision records
- API documentation
- Wikis

This evidence explains what the repository explicitly communicates.

---

## Historical Evidence

Evidence preserved through repository history.

Examples include:

- Commit history
- Release notes
- Changelogs
- Version history
- Tags

This evidence explains how the repository evolved.

---

## Development Evidence

Evidence produced through implementation activities.

Examples include:

- Pull requests
- Issues
- Code reviews
- Discussions
- Merged changes

This evidence explains how decisions became implementation.

---

## Structural Evidence

Evidence found within the repository itself.

Examples include:

- Directory structure
- Project organization
- Module relationships
- Configuration
- Build systems

This evidence explains how the repository is organized.

---

## Governance Evidence

Evidence describing project management.

Examples include:

- Contribution policies
- Security policies
- Codes of conduct
- Maintainer documentation
- Governance processes

This evidence explains how the project operates.

---

## Community Evidence

Evidence created through contributor interaction.

Examples include:

- Review conversations
- Discussion threads
- Contributor activity
- Collaboration history
- Community practices

This evidence explains how people work together.

---

# Evidence Characteristics

Useful evidence should be:

- Relevant
- Traceable
- Verifiable
- Contextual
- Current
- Connected

Evidence should always point back to its original repository artifact.

ORI should never generate evidence that cannot be inspected.

---

# Evidence Relationships

Evidence gains meaning when considered together.

Examples include:

- Documentation may support a historical decision.
- A pull request may explain a commit.
- Governance may influence contributor activity.
- A release note may summarize several repository changes.

Repository Intelligence is strengthened by connected evidence rather than isolated artifacts.

---

# Evidence Confidence

Not every piece of evidence carries equal weight.

Confidence reflects how strongly available evidence supports an observation.

Confidence should increase when:

- Multiple independent artifacts support the same conclusion.
- Evidence is recent and well maintained.
- Evidence is internally consistent.
- Evidence can be traced directly to repository artifacts.

Confidence should decrease when:

- Evidence is incomplete.
- Evidence conflicts.
- Evidence is outdated.
- Evidence depends heavily on inference.

Confidence reflects the quality of supporting evidence, not certainty.

---

# Evidence Before Insight

ORI follows a simple principle:

```
Repository Artifact

↓

Evidence

↓

Observation

↓

Insight

↓

Recommendation
```

Recommendations should never bypass evidence.

Every recommendation should remain explainable through the observations and evidence that produced it.

---

# Explainability

Every observation generated by ORI should answer four questions.

## What was observed?

The repository pattern that was identified.

---

## Why does it matter?

The repository context that gives the observation significance.

---

## What evidence supports it?

The repository artifacts that justify the observation.

---

## Where can it be verified?

The original repository locations where maintainers can review the supporting evidence themselves.

Explainability is essential to Repository Intelligence.

Understanding should never depend on trusting a hidden process.
